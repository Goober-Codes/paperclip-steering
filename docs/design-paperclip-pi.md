## Problem Statement

Run a real fashion brand using an augmented Paperclip stack (Pi agent + RPC steering +
kanban board) as the operational backbone. Tasks: content creation, supplier/ops
coordination, market/trend research, customer comms.

Simultaneously: make upstream contributions to Paperclip at each phase, sequenced by
COMPANY.md. Each PR maps to a scalability concept from *Web Scalability for Startup
Engineers*. The fashion brand is the test case. Merged PRs are the public artifact.

---

## What Makes This Cool

Paperclip is the only orchestration platform where a human can redirect a running agent
mid-task without killing the process. Pi is the only agent runtime with a structured
steering protocol. Nobody is running a fashion brand on this stack and writing about it.

The learning strategy (seen in ColeMurray/background-agents contributor history):
learn a concept by making upstream contributions to a real production repo, sequenced
by a self-written phase plan, with merged PRs as the public artifact. Not a tutorial
project. Each PR is both a product improvement and a scalability concept made concrete.

(Out of scope for this roadmap: multi-agent creative debate chamber. Revisit after Phase 3.)

---

## Constraints

- Pi's tool coverage: read/write/bash/edit only. No browser, no email client. Fashion
  tasks (supplier emails, social media posting) need either: (a) Pi bash tools shelling
  out to CLI email/social tools, or (b) skills/plugins that extend coverage.
- RPC steering is currently Phase 3 in COMPANY.md but is the core thesis — demo it early.
- Kanban board is developer-native. Fashion brand operator UI may need a simpler view.
- Scalability concepts (Ch 7 async, Ch 3 stateless, Ch 4 SOA) are load-bearing design
  constraints, not afterthoughts — even at single-brand scale.

---

## Premises (Agreed)

1. Pi's tool coverage is insufficient for all fashion tasks as-is — needs skill extension
2. RPC steering is the core differentiator — should be demoed before Phase 3
3. Content and supplier tasks have different SLAs — may need separate queue treatment
4. Kanban board is the right interface for this operator (developer mindset)
5. Scalability engineering is partly educational scaffolding — apply it deliberately

Coolest version not yet considered: multi-agent creative debate chamber. Fire a content

Weekend MVP identified (unverified — read rpc.md first to confirm Pi accepts steer
mid-flight before coding anything):
- In `execute.ts`: add `mode: "rpc"` code path. Call `spawn("pi", ["--mode", "rpc",
  "--session", path])` directly (not via `runChildProcess` in adapter-utils, which
  doesn't expose raw stdin). Store `ChildProcess` in a `Map<runId, ChildProcess>`
  singleton in the adapter.
- New route in `server/src/routes/`: `POST /api/runs/:runId/steer` — looks up the
  live process by runId, writes `{"type":"steer","message":"..."}` to its stdin.
  Request body: `{ message: string }`. Response: `{ ok: true }` or 404 if run not found.
- In UI (`ui/src/components/IssueDetail.tsx` or equivalent): add a "Steer" text input +
  button below the comment box. Calls the new endpoint. No new views needed.
- Cleanup: on run completion or error, delete from the Map and close stdin.

**Decision on Open Question 1:** Comment box = `follow_up` (delivered when agent
finishes). Steer button = `steer` (delivered mid-flight). Two distinct affordances,
both wired to the RPC pipe. This is decided — proceed with implementation.

Inngest flagged as architectural reference for the async step graph — not a replacement,
but worth reading their `waitForEvent` pause/resume model before touching execute.ts.

## Recommended Approach

**Approach B — Upstream contributor.**

Follow COMPANY.md phases but reframe each phase as a PR:

| Phase | COMPANY.md goal | PR target | Scalability concept applied |
|-------|----------------|-----------|---------------------|
| 0 (pre-work) | Understand the codebase | None — read only | Ch 2: understand design before touching. Draw the architecture diagram. |
| 1 | Weekend MVP + first PR | Doc fix or small bug in pi_local (find one during setup). PR 1 is small by design — establish contrib relationship with maintainers. | Ch 2: single responsibility, loose coupling |
| 2 | HITL workflows | Improve HITL documentation in `docs/` OR add a missing UI affordance found during testing | Ch 7: async processing — the heartbeat scheduler is a message queue. Document it as such. |
| 3 | RPC steering | `mode: "rpc"` in execute.ts + `POST /api/runs/:runId/steer` + Steer UI button. Feature-flagged. | Ch 3: stateless adapter (stdin pipe is per-run, not per-adapter-instance). Ch 4: service isolation. |
| 4 | Polish + template | Fashion brand example company config (JSON) committed to `examples/`. COMPANY.md as contributor guide in `docs/`. | Ch 9: CI for the adapter (add tests to the Phase 3 PR). |

**Timeline clarification:** Phase 3 (full upstream PR) is a 4-6 week effort including
tests, docs, and review cycles. The "weekend MVP" is a local proof-of-concept on a
private branch — it demonstrates the thesis but is not PR-ready. The weekend build
gates the Phase 3 PR: only open it after the demo has run a real task end-to-end.

**Known limitation:** `Map<runId, ChildProcess>` is in-memory per process. If the
Paperclip server restarts while a run is in-flight, the steer endpoint returns 404 for
that run. Acceptable for local dev and the upstream PR (document it in the PR description
as a known limitation). Persistence across restarts is out of scope.

## What I noticed about how you think

- You said "I'm running a fashion brand via this augmented paperclip" — not "I want to
  build a tool for fashion brands." You're the user. That's a meaningful difference.
  Builders who use their own tools find the real problems faster.

- You picked "non-technical founders" as the demo audience, then described a kanban
  board full of agent tasks. Those two things don't fully reconcile yet — and you know it
  (you agreed with Premise 4). The tension between "I think like a dev" and "I want to
  show this to non-technical people" is worth sitting with.

- You kept the scalability learning goal on the table even when the conversation moved
  toward product vision. That's unusual and deliberate. You're not just building — you're
  building with a curriculum. That's a serious way to learn.

- When asked about the fastest demo path, you pivoted entirely from the RPC plumbing to
  "a kanban view for fashion brand tasks." That's product instinct: skip the tech, show
  the use case. The subagent caught it too — "fleets" was the tell.
